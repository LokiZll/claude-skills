# 设计与规划

将想法转化为完整的设计和实施计划。在编写任何代码之前，先研究现有方案、探索需求、验证设计。

## 核心原则

1. **研究优先** — 先搜索现有解决方案，避免重复造轮子
2. **设计先行** — 在获得用户批准前不写代码
3. **增量验证** — 每个阶段都获得用户确认

## 完整工作流

```
研究 → 理解 → 设计 → 计划 → 实施

1. 研究：搜索现有方案（npm/PyPI/MCP/GitHub）
2. 理解：一次一个问题澄清需求
3. 设计：提出 2-3 种方法，获得批准
4. 计划：创建详细实施步骤
5. 实施：按计划编码
```

---

## 阶段 1：研究现有方案

在写任何代码之前，先检查是否已有解决方案。

### 快速检查清单

0. **仓库内搜索** → `rg` 搜索相关模块/测试
1. **包管理器** → npm/PyPI 搜索
2. **MCP 服务器** → 检查 `~/.claude/settings.json`
3. **Skills** → 检查 `~/.claude/skills/`
4. **GitHub** → 代码搜索找参考实现

### 决策矩阵

| 信号 | 行动 |
|------|------|
| 完全匹配，维护良好 | **采用** — 直接安装使用 |
| 部分匹配，基础良好 | **扩展** — 安装 + 薄包装器 |
| 多个弱匹配 | **组合** — 结合 2-3 个小包 |
| 没找到合适的 | **构建** — 写自定义代码 |

### 常用工具速查

| 类别 | 工具 |
|------|------|
| Linting | eslint, ruff, textlint |
| 格式化 | prettier, black |
| 测试 | jest, pytest, playwright |
| HTTP | httpx (Python), ky/got (Node) |
| 验证 | zod (TS), pydantic (Python) |
| Markdown | remark, unified |

## 阶段 2：理解需求

通过对话澄清需求，一次一个问题。

### 检查清单

1. **探索项目背景** — 检查文件、文档、最近提交
2. **提出澄清问题** — 了解目的、约束、成功标准
3. **识别边缘情况** — 错误处理、空状态、并发

### 提问原则

- 一次一个问题，不要淹没用户
- 首选选择题，比开放式更容易回答
- 专注于：目的、约束、成功标准

---

## 阶段 3：设计方案

提出 2-3 种方法，获得用户批准后再继续。

### 设计内容

- 架构概述
- 组件划分
- 数据流
- 错误处理
- 测试策略

### 呈现方式

- 以推荐方案开头，解释原因
- 列出每种方案的权衡
- 根据复杂度调整详细程度
- 每节后询问是否正确

### 设计文档

批准后保存到 `docs/plans/YYYY-MM-DD-<主题>-design.md`

## 阶段 4：实施计划

将设计分解为可执行的步骤。

### 计划格式

```markdown
# 实施计划：[功能名称]

## 概述
[2-3 句摘要]

## 需求
- [需求 1]
- [需求 2]

## 架构更改
- [文件路径：描述]

## 实施步骤

### 阶段 1：[阶段名称]
1. **[步骤名称]** (文件: path/to/file.ts)
   - 操作：具体行动
   - 原因：为什么
   - 依赖：无 / 需要步骤 X
   - 风险：低/中/高

## 测试策略
- 单元测试：[文件]
- 集成测试：[流程]
- E2E 测试：[用户旅程]

## 成功标准
- [ ] 标准 1
- [ ] 标准 2
```

### 分阶段原则

- **阶段 1**：最小可行 — 提供价值的最小切片
- **阶段 2**：核心体验 — 完整的快乐路径
- **阶段 3**：边缘情况 — 错误处理、打磨
- **阶段 4**：优化 — 性能、监控

每个阶段应可独立合并。

## 示例：添加用户认证

### 研究阶段
```
搜索：npm "nextjs authentication"
发现：next-auth (评分：9/10)
决策：采用 — 直接使用 next-auth
```

### 理解阶段
```
Q: 需要支持哪些登录方式？
A: Google + GitHub OAuth

Q: 需要存储用户数据吗？
A: 是，用 Prisma + PostgreSQL
```

### 设计阶段
```
方案 A（推荐）：next-auth + Prisma adapter
- 优点：官方支持，文档完善
- 缺点：配置稍复杂

方案 B：自定义 JWT
- 优点：完全控制
- 缺点：需要自己处理安全问题
```

### 计划阶段
```
阶段 1：基础认证
1. 安装 next-auth 和 prisma adapter
2. 配置 Google OAuth provider
3. 创建 User 模型

阶段 2：完善体验
4. 添加 GitHub provider
5. 创建登录/登出 UI
6. 添加 session 中间件
```

## 反模式

- **直接写代码** — 不研究就开始实现
- **跳过设计** — "这太简单了，不需要设计"
- **忽略现有方案** — 重复造轮子
- **一次问太多** — 用多个问题淹没用户
- **计划太粗** — 没有具体文件路径和步骤

## 关键原则

- **YAGNI** — 删除不必要的功能
- **最小更改** — 优先扩展现有代码
- **保持模式** — 遵循项目约定
- **支持测试** — 构建易于测试的更改
- **增量交付** — 每个阶段可独立合并
